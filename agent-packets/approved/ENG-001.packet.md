## Guardrails

### 01-system-architecture.s005 (`@optcg/engine-core`)
Server-only authoritative game engine.

Contains:

- Full `GameState`, including hidden zones and RNG state.
- `applyAction`, `getLegalActions`, `resumeDecision`.
- Rule processing, battle flow, effect queue, event journal.
- Full visibility filtering functions.
- State hash generation.

Does not contain:

- React components.
- Database queries.
- WebSocket room management.
- Poneglyph HTTP calls.

### 03-game-state-events-decisions.s020 (State hashing)
Hashing must be deterministic for the same canonical state contents. Avoid host-dependent ordering and avoid transient process data in the hash input.

### 03-game-state-events-decisions.s021 (Invariant hooks)
Invariant hooks run after accepted actions and decision resumes in dev/test flows. They are part of the engine-core safety boundary and should fail closed when the state becomes inconsistent.

### 11-testing-quality.s008 (Invariant tests)
Run after every action, decision response, effect resolution, and replay step in test mode.

```ts
assertAllCardsInExactlyOneLocation(state);
assertNoDuplicateInstanceIds(state);
assertCharacterAreaSizeAtMostFive(state);
assertStageAreaSizeAtMostOne(state);
assertLeaderAreaExactlyOne(state);
assertAttachedDonConsistency(state);
assertNoIllegalHiddenInfoInViews(state);
assertPendingDecisionIsValid(state);
assertEffectQueueEntriesAreResolvableOrCancelled(state);
assertStateHashStable(state);
```

## Fuzz tests

## Scope
- create packages/engine-core with the canonical createInitialState, getLegalActions, applyAction, resumeDecision, computeView, filterStateForPlayer, and hashGameState exports
- establish a deterministic server-only GameState / EngineResult skeleton with state sequencing, event journal plumbing, and hidden-zone-safe filtering entry points
- implement deterministic hashGameState and baseline filterStateForPlayer behavior for the canonical contracts
- wire invariant assertions into engine test mode and preserve the server-only boundary for authoritative state handling

## Out of Scope
- deterministic setup, opening-hand, mulligan, and life-placement flow (ENG-007)
- turn-engine first-turn restrictions and draw/DON phase semantics beyond the bootstrap boundary (ENG-010)
- DON attach/return mechanics (ENG-011)
- non-mulligan PendingDecision runtime and generic decision progression (ENG-015)
- main-phase play/attach/attack legality and vanilla combat/damage completion flows (ENG-012, ENG-013)
- full non-vanilla effect runtime behavior
- websocket room management
- browser-safe presentation helpers
