# Story Packet

## Story

Spec Version: v6
Story Schema Version: 1.0.0
ID: ENG-016
Title: Modularize engine-core bootstrap internals and contract tests
Type: implementation
Area: engine

## Why
The bootstrap engine surface is now concentrated in one large implementation file and one large contract test file. That layout is increasing review churn, making subsystem boundaries harder to reason about, and raising the risk of regressions when narrow fixes land in unrelated logic. This story performs a behavior-preserving structural cleanup after ENG-001.

### 01-system-architecture.s005 (`@optcg/engine-core`)
Server-only authoritative game engine.

Contains:

- Full `GameState`, including hidden zones and RNG state.
- `applyAction`, `getLegalActions`, `resumeDecision`.
- Rule processing, battle flow, effect queue, event journal.
- Full visibility filtering functions.
- State hash generation.

Does not contain:

- React components.
- Database queries.
- WebSocket room management.
- Poneglyph HTTP calls.

### 03-game-state-events-decisions.s020 (State hashing)
Hashing must be deterministic for the same canonical state contents. Avoid host-dependent ordering and avoid transient process data in the hash input.

### 03-game-state-events-decisions.s021 (Invariant hooks)
Invariant hooks run after accepted actions and decision resumes in dev/test flows. They are part of the engine-core safety boundary and should fail closed when the state becomes inconsistent.

### 11-testing-quality.s008 (Invariant tests)
Run after every action, decision response, effect resolution, and replay step in test mode.

```ts
assertAllCardsInExactlyOneLocation(state);
assertNoDuplicateInstanceIds(state);
assertCharacterAreaSizeAtMostFive(state);
assertStageAreaSizeAtMostOne(state);
assertLeaderAreaExactlyOne(state);
assertAttachedDonConsistency(state);
assertNoIllegalHiddenInfoInViews(state);
assertPendingDecisionIsValid(state);
assertEffectQueueEntriesAreResolvableOrCancelled(state);
assertStateHashStable(state);
```

## Scope
- split `packages/engine-core/src/engine-core.ts` into focused internal modules for invariants, visibility filtering, pending decisions, legal actions, state hashing, and mutation logic
- split `tests/infra/engine-core-api.test.ts` into focused engine-core contract test files by concern
- preserve the current public package entrypoint and approved ENG-001 bootstrap behavior
- add or retain regression coverage needed to prove the refactor is behavior-preserving

## Out of Scope
- new gameplay actions, combat flow, setup/runtime expansion, or pending-decision progression
- canonical type or protocol changes
- package renames or wider architecture work outside `@optcg/engine-core` and its direct infra tests

## Follow-On Chain
- `ENG-001`: bootstrap the deterministic engine-core API skeleton
- `ENG-007`: deterministic setup, opening hands, mulligan, life placement
- `ENG-010`: phase progression and first-turn restrictions
- `ENG-011`: DON!! attach/return mechanics
- `ENG-015`: generic pending-decision runtime beyond the bootstrap mulligan path
- `ENG-012`: main-phase play actions and event-card skeleton
- `ENG-013`: vanilla attack, damage, K.O., and terminal combat flows

## Constraints
- refactor only; do not invent or widen runtime behavior
- preserve hidden-information safety and deterministic behavior across extracted modules
- keep the public engine-core package surface stable

## Required Tests
- contract regression coverage for `hashGameState`, `filterStateForPlayer`, `getLegalActions`/`applyAction`, `resumeDecision`, and invariant/test-mode behavior after the refactor
- package-entrypoint smoke test proving the published `@optcg/engine-core` surface still builds and resolves

## Expected Output
- internal module extraction under `packages/engine-core/src/`
- split infra tests for engine-core contracts
- no behavior change to the approved bootstrap surface
- brief implementation note explaining the new internal module boundaries

## Acceptance Criteria
- [ ] The `@optcg/engine-core` public entrypoint exports and import paths remain stable for canonical consumers.
- [ ] `engine-core.ts` becomes a thin orchestration/export layer with responsibility-specific logic moved into focused internal modules.
- [ ] The current monolithic engine-core API contract test file is split into focused test files without loss of bootstrap coverage.
- [ ] `pnpm verify` passes with no behavior drift in the approved bootstrap engine-core surface.

## Ambiguity Rule
If the refactor would require behavior changes, contract changes, or speculative cleanup outside the approved bootstrap surface, stop and escalate instead of widening scope.
