## Relevant Spec Excerpts
### 03-game-state-events-decisions.s019 (Deterministic RNG)
The engine must never use `Math.random()`.

```ts
interface RngState {
  algorithm: 'pcg32' | 'xoshiro256ss' | 'test-fixed';
  seedCommitment?: string;
  internalState: string;
  callCount: number;
}

interface RngDrawResult<T> {
  value: T;
  nextRng: RngState;
  event: EngineEvent;
}
```

All shuffle operations emit an event without exposing the resulting order to players.

### Hidden information is a security boundary

### 02-engine-mechanics.s008 (Setup sequence)
Use a deterministic setup flow:

```text
1. Validate decks, card support, format, banlist.
2. Determine first/second player.
3. Build initial full GameState.
4. Resolve start-of-game effects that modify setup.
5. Shuffle decks using recorded RNG.
6. Draw opening hands.
7. Handle official mulligan decisions: each player may once either keep or return all 5 to deck, reshuffle, and redraw 5, in first-player-then-second-player order.
8. Place Life from the top of deck equal to Leader life using the canonical orientation algorithm below.
9. Start first player's Refresh Phase.
```

Start-of-game effects that alter decks must happen before final shuffle and opening draw.

### 03-game-state-events-decisions.s005 (Event journal)

### 03-game-state-events-decisions.s006 (Event visibility)
Events may contain hidden data. Filter them before sending to clients.

```ts
type EventVisibility =
  | { type: 'public' }
  | { type: 'private'; playerId: PlayerId }
  | { type: 'hidden' }
  | { type: 'replayOnly' };
```

## Atomic mutation contract

## Scope
- Implement RngState model with algorithm, internalState, callCount (and optional seedCommitment).
- Add deterministic RNG step function(s) for supported algorithms (pcg32, xoshiro256ss, test-fixed) and a typed draw API returning nextRng + EngineEvent.
- Implement deterministic shuffle(list, rng) -> {shuffled,nextRng,event} used by setup sequence.
- Ensure shuffle emits an EngineEvent with visibility that does not expose resulting order to players (payload must not contain full ordered card IDs for secret zones).

## Out of Scope
- Cryptographic commitments/seed commitment verification beyond storing seedCommitment if present.
- Match-server seed negotiation or anti-cheat beyond engine guarantees.
- Public-view hashing strategy (separate story).

## Constraints
- All RNG state must be stored in GameState.rng and remain JSON-serializable.
- Events for shuffles must not leak secret order.

## Required Tests
- Unit: RNG draw determinism for each supported algorithm (golden vectors or snapshot).
- Unit: shuffle determinism and permutation validity.
- Static/lint/test: fail if Math.random is referenced in engine-core package.
