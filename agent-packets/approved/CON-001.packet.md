## Why

### 3. DSL schema

### 01-system-architecture.s004 (`@optcg/types`)
Shared TypeScript types with no runtime dependencies on app packages.

Contains:

- `CardId`, `InstanceId`, `PlayerId`, `MatchId` branded types.
- Public action and message contracts.
- `PlayerView`, `SpectatorView`, and replay DTOs.
- Card-data schemas.
- Deck validation input/output types.

Does not contain:

- Full engine implementation.
- Database clients.
- WebSocket clients.
- Browser-specific code.

### `@optcg/engine-core`

## Scope
- create packages/types seeded from contracts/canonical-types.ts
- export the canonical foundation contracts that are already concretely specified in the current contract/spec set, including branded ids, Action, PendingDecision, PlayerView, replay/card-manifest types, and card metadata interfaces
- make @optcg/types a real built workspace package with a package-local entrypoint and declaration surface
- keep the package free of runtime dependencies on engine, database, websocket, browser, or sibling workspace code
- wire contract drift, build-layout, and package-boundary checks into repo verification

## Out of Scope
- finalizing public protocol DTOs such as ClientActionEnvelope, ServerMessage, or filtered PublicDecision payloads
- defining or freezing SpectatorView shape or spectator-mode-specific payloads
- finalizing deck-validation input/output DTOs
- full engine behavior
- match-server transport handling
- browser view helpers

## Constraints
- must pass pnpm verify
- must preserve package boundary enforcement
- must not introduce hidden-information leakage

## Required Tests
- contract compile and package-build check covering contracts/canonical-types.ts and packages/types through the repo toolchain
- import-boundary regression proving sibling package and app imports are not pulled into packages/types
- API-surface regression covering the narrowed foundation exports only
