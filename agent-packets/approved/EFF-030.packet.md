## Source presence policy

### 04-effect-runtime.s008 (Trigger detection from events)
Trigger detection consumes event batches.

```ts
function detectTriggeredEffects(
  state: GameState,
  events: EngineEvent[]
): TriggerCandidate[] {
  const candidates: TriggerCandidate[] = [];

  for (const event of events) {
    candidates.push(...findAutoEffectsForEvent(state, event));
    candidates.push(...findReplacementFollowupsIfAny(state, event));
  }

  return candidates.filter(c => canTriggerNow(c, state));
}
```

The engine must check source presence before queueing, then apply the queue entry's source-presence policy before resolution.

## Queue ordering

### 04-effect-runtime.s009 (Queue ordering)
Every trigger collection creates or joins a timing window. Queue order is deterministic and must not depend on JavaScript array discovery order except where the spec explicitly says discovery order is the canonical tie-breaker.

Normative ordering algorithm:

```text
1. Assign every collected trigger a timingWindowId.
2. Assign generation = 0 for effects triggered by the original timing event.
3. When resolving an effect produces new triggers, enqueue them with generation = currentGeneration + 1 in the same timing window unless a new official timing window has opened.
4. Resolve older timing windows before newer timing windows.
5. Within a timing window, resolve lower generation before higher generation.
6. Within a generation, resolve turn-player bucket before non-turn-player bucket.
7. Within a player's bucket, if more than one effect is pending, create chooseTriggerOrder for that player.
8. If no choice is required, use stable tie-breakers: createdAtEventSeq, then source instance id, then effect id.
```

Consequences:

- If turn player effect A and non-turn player effect B are pending, and A creates turn player effect C while resolving, B resolves before C.
- Effects triggered during damage processing wait until all damage points are complete, except `[Trigger]` resolution itself.
- Effects triggered during an effect or card activation wait until the triggering process completes.
- Optional triggered effects create `chooseOptionalActivation` decisions at the point they would enter or begin resolution, according to the card's timing rule.

## Scope
- Define and persist EffectQueueEntry fields on GameState (id/state/timingWindowId/generation/controllerId/source/sourceSnapshot/triggerEventId/effectBlockId/orderingGroup/createdAtEventSeq/queuedAtStateSeq/sourcePresencePolicy/causedBy).
- Implement enqueueTriggeredEffectsRespectingTiming() to assign timingWindowId and generation per normative algorithm.
- Implement deterministic dequeueEffect() that respects ordering rules (window->generation->turnPlayer bucket->nonTurnPlayer bucket->choice or stable tie-breakers).
- Create PendingDecision.chooseTriggerOrder when a player has >1 pending effect in their bucket within a generation.
- Implement stable tie-breakers: createdAtEventSeq, then source instance id, then effect id.

## Out of Scope
- Executing specific DSL effect primitives.
- Replacement effects processing (handled separately).
- Continuous effect computed view application (handled separately).

## Constraints
- Effect ordering must not depend on JS discovery/iteration order except where explicitly specified (04-effect-runtime.s009).

- Unit: simultaneous triggers controlled by same player produce a chooseTriggerOrder decision (18-acceptance-tests.s004 M2-008).
- Property/golden: repeated runs on same inputs produce identical dequeue sequence and state hashes.

## Acceptance Criteria
- [ ] Effect triggers collected from an event batch are assigned a timingWindowId and generation=0 for the original timing event (04-effect-runtime.s009).
- [ ] New triggers produced during effect resolution are enqueued with generation=current+1 in the same timing window unless a new timing window opens (04-effect-runtime.s009).
- [ ] Within a timing window, lower generation resolves before higher generation; within a generation, turn-player bucket resolves before non-turn-player bucket (04-effect-runtime.s009).
- [ ] When a player has multiple pending effects in their bucket, the engine pauses with chooseTriggerOrder and resumes deterministically (04-effect-runtime.s009).
- [ ] If no choice is required, ordering is deterministic using the specified tie-breakers (04-effect-runtime.s009).
- [ ] Satisfies ordering-related Milestone 2 acceptance items M2-008 and M2-009 (18-acceptance-tests.s004).
