### 02-engine-mechanics.s010 (Turn structure)
Turn player may repeatedly:

- Play a Character, Stage, or `[Main]` Event from hand.
- Activate `[Activate: Main]` effects.
- Give active DON!! to Leader or Characters.
- Declare an attack, if legal.
- End the phase.

### 03-game-state-events-decisions.s016 (Action envelope inside the engine)
```ts
type Action =
  | { type: 'playCard'; cardInstanceId: InstanceId; costPayment?: PaymentSpec }
  | { type: 'activateEffect'; source: CardRef; effectId: string; costPayment?: PaymentSpec }
  | { type: 'attachDon'; donInstanceId: InstanceId; target: CardRef }
  | { type: 'declareAttack'; attacker: CardRef; target: CardRef }
  | { type: 'activateBlocker'; blocker: CardRef }
  | { type: 'useCounter'; cardInstanceId: InstanceId; target: CardRef }
  | { type: 'endMainPhase' }
  | { type: 'concede'; playerId: PlayerId }
  | { type: 'respondToDecision'; decisionId: string; response: DecisionResponse };
```

### 03-game-state-events-decisions.s017 (Canonical decision routing)
All player choices are represented as `PendingDecision` and answered by exactly one action shape:

```ts
{ type: 'respondToDecision', decisionId, response }
```

The engine validates the response against the current pending decision. The client never gets to submit raw target IDs or payment choices outside the active decision context.

## Scope
- Implement `Action.playCard` legality during Main Phase for Character and Stage cards in hand, using authoritative state and cost availability.
- Integrate implemented play flows with PendingDecision/payment routing from `ENG-015` where a choice is required to pay or select cards.
- Move played Characters into `characterArea` with `turnPlayed` metadata and enforce character-area size limits.
- Move played Stages into `stageArea` with stage-area legality checks.
- Implement an explicit event-card skeleton path: legal to play when costs can be paid, emits canonical events, and resolves through a no-effect / unsupported placeholder without inventing full effect runtime.

## Out of Scope
- Full effect DSL execution or card-specific abilities.
- Attack resolution, damage, or K.O. handling.
- Counter timing and blocker/counter battle windows.

## Constraints
- Legal play actions must be derived from authoritative state, not view hints.
- Do not bypass pending-decision routing for costs or selections that require player choice.

## Required Tests
- Integration: play a Character in Main Phase and assert cost payment, zone move, `turnPlayed` metadata, and emitted events.
- Integration: reject an illegal second Stage while one is already present.
- Integration: event-card skeleton path emits canonical events and ends in an explicit placeholder/no-effect state without mutating unrelated state.

## Acceptance Criteria
- [ ] `getLegalActions` exposes play-card actions during the active player's Main Phase when a Character or Stage can legally be played.
- [ ] Applying a legal Character or Stage play updates zones, costs, and state sequencing authoritatively and emits canonical events.
- [ ] Character and stage area limits remain enforced after plays.
- [ ] Event-card skeleton actions do not invent runtime effects; they resolve through an explicit placeholder path that preserves sequencing and event emission.
