# Match Server Protocol
<!-- SECTION_REF: 07-match-server-protocol.s001 -->
Section Ref: `07-match-server-protocol.s001`

## Purpose
<!-- SECTION_REF: 07-match-server-protocol.s002 -->
Section Ref: `07-match-server-protocol.s002`

The match server hosts live games over WebSocket. It is authoritative, stateful, and responsible for action sequencing, timers, reconnection, and filtered state delivery.

The engine owns rules. The match server owns transport and session behavior. Protocol data structures should use the ID and action types from [`contracts/canonical-types.ts`](contracts/canonical-types.ts). Queue entry and custom-lobby creation are defined separately in [`29-game-types-queues-and-lobbies.md`](29-game-types-queues-and-lobbies.md); this document begins once a match already exists.

## Connection lifecycle
<!-- SECTION_REF: 07-match-server-protocol.s003 -->
Section Ref: `07-match-server-protocol.s003`

```text
client connects
  -> authenticate session
  -> join match room
  -> server sends current PlayerView/SpectatorView
  -> client sends action envelopes
  -> server validates and applies through engine
  -> server broadcasts filtered results
```

## Client action envelope
<!-- SECTION_REF: 07-match-server-protocol.s004 -->
Section Ref: `07-match-server-protocol.s004`

Every client action is idempotent and sequence-aware.

```ts
interface ClientActionEnvelope {
  protocolVersion: string;
  matchId: MatchId;
  playerId: PlayerId;
  clientActionId: string; // UUID generated by client
  expectedStateSeq: number;
  expectedDecisionId?: string;
  actionHash: string;       // canonical hash of action only, excluding transport metadata
  sentAtClientTime?: string; // audit only
  action: Action;
  signature?: string;       // disabled for MVP unless signature rules are explicitly configured
}
```

The server stores recent `clientActionId`s per match/player to dedupe reconnect resends and double-clicks. Idempotency is keyed by `(matchId, playerId, clientActionId)`.

Idempotency behavior:

| Condition | Server behavior |
|---|---|
| New `clientActionId` and matching `expectedStateSeq` | Validate and apply. |
| Repeated `clientActionId` with same `actionHash` | Return the previously stored result without applying again. |
| Repeated `clientActionId` with different `actionHash` | Reject with `idempotencyConflict`. |
| Lower `expectedStateSeq` with no idempotency match | Reject with `staleState`. |
| Higher `expectedStateSeq` | Reject with `futureState` or hold only if an explicit server queue policy is later added. |

## Server action result
<!-- SECTION_REF: 07-match-server-protocol.s005 -->
Section Ref: `07-match-server-protocol.s005`

```ts
interface ServerActionResult {
  type: 'actionResult';
  matchId: MatchId;
  clientActionId: string;
  accepted: boolean;
  stateSeq: number;
  actionSeq: number;
  reason?: RejectionReason;
  view?: PlayerView;
  events?: PublicEffectEvent[];
}

type RejectionReason =
  | 'staleState'
  | 'futureState'
  | 'idempotencyConflict'
  | 'notYourTurn'
  | 'illegalAction'
  | 'pendingDecisionMismatch'
  | 'rateLimited'
  | 'matchFrozen'
  | 'unsupportedCard'
  | 'serverError';
```

## Broadcast messages
<!-- SECTION_REF: 07-match-server-protocol.s006 -->
Section Ref: `07-match-server-protocol.s006`

```ts
type ServerMessage =
  | ServerActionResult
  | { type: 'stateSync'; matchId: MatchId; serverSeq: number; stateSeq: number; view: PlayerView }
  | { type: 'decisionRequired'; matchId: MatchId; serverSeq: number; stateSeq: number; decision: PublicDecision }
  | { type: 'timerUpdate'; matchId: MatchId; serverSeq: number; stateSeq: number; timers: PublicTimerState }
  | { type: 'opponentDisconnected'; matchId: MatchId; serverSeq: number; timeoutAt: string }
  | { type: 'opponentReconnected'; matchId: MatchId; serverSeq: number }
  | { type: 'matchError'; matchId: MatchId; serverSeq: number; message: string; reportToken: string }
  | { type: 'matchEnded'; matchId: MatchId; serverSeq: number; stateSeq: number; result: MatchResult }
  | { type: 'ping'; serverTime: string }
  | { type: 'pong'; serverTime: string };
```

Messages are per-recipient filtered. Two players may receive different payloads for the same state sequence. Every non-heartbeat server message has a monotonically increasing `serverSeq` per recipient connection and includes `stateSeq` when tied to game state. Clients discard messages older than their latest accepted `serverSeq` or `stateSeq`.

## Action application flow
<!-- SECTION_REF: 07-match-server-protocol.s007 -->
Section Ref: `07-match-server-protocol.s007`

```ts
async function handleClientAction(conn, envelope) {
  authenticateEnvelope(conn, envelope);
  assertPlayerInMatch(conn.playerId, envelope.matchId);

  const prior = lookupIdempotency(envelope.matchId, conn.playerId, envelope.clientActionId);
  if (prior && prior.actionHash === envelope.actionHash) return prior.result;
  if (prior && prior.actionHash !== envelope.actionHash) return reject(envelope, 'idempotencyConflict');

  assertExpectedState(envelope.expectedStateSeq);
  assertRateLimit(conn.playerId, envelope.action.type);

  const state = match.currentState;
  const legal = engine.getLegalActions(state, conn.playerId);

  if (!isLegal(envelope.action, legal)) {
    logRejectedAction(envelope, state.stateSeq);
    const rejected = reject(envelope, 'illegalAction');
    storeIdempotency(envelope, rejected);
    return rejected;
  }

  const result = envelope.action.type === 'respondToDecision'
    ? engine.resumeDecision(state, envelope.action.response)
    : engine.applyAction(state, envelope.action);

  match.currentState = result.state;
  match.appendDeterministicReplayEntry(envelope.action, result);
  match.appendAuditEnvelope(envelope);
  match.persistActionSinceSnapshot(envelope);

  const actionResult = buildActionResult(envelope, result);
  storeIdempotency(envelope, actionResult);

  sendActionResult(conn, actionResult);
  sendFilteredViewsToPlayers(match, result);
  sendSpectatorUpdates(match, result);
}
```

## Broadcast ordering
<!-- SECTION_REF: 07-match-server-protocol.s008 -->
Section Ref: `07-match-server-protocol.s008`

For each accepted action or decision response, commit state before broadcasting. The broadcast order for each recipient is:

1. `actionResult` to the acting connection, accepted or rejected.
2. `stateSync` with the recipient's filtered `PlayerView`, if state changed.
3. `decisionRequired`, if the resulting state is paused on a decision visible to that recipient.
4. `timerUpdate`, if timer ownership or remaining time changed.
5. `matchEnded`, if the game completed.

All messages in this sequence carry increasing `serverSeq`. `stateSync`, `decisionRequired`, `timerUpdate`, and `matchEnded` include the committed `stateSeq`. Animation/event feeds may be embedded in `stateSync` or sent later, but they cannot overtake the committed state message they describe.

`signature` is not used in MVP. If enabled later, the signed payload is the canonical serialized `action` plus `matchId`, `playerId`, `clientActionId`, `expectedStateSeq`, and `actionHash`; it excludes `sentAtClientTime` and transport metadata.

## State sync
<!-- SECTION_REF: 07-match-server-protocol.s009 -->
Section Ref: `07-match-server-protocol.s009`

Use full `PlayerView` sync after meaningful state transitions. Diffs are optional later optimization.

Early implementation should prefer correctness:

- Send full filtered view to each player after accepted action/decision.
- Include `stateSeq`.
- Client discards stale views.
- Client can request resync.

## Decision protocol
<!-- SECTION_REF: 07-match-server-protocol.s010 -->
Section Ref: `07-match-server-protocol.s010`

When the engine pauses:

1. Server sends `decisionRequired` only to the required player.
2. Server may send public "waiting" status to the other player.
3. Required player responds with `respondToDecision` action in a normal envelope.
4. Server validates `expectedDecisionId`.
5. Engine resumes.

There is no separate decision-default auto-response in competitive play. If a player is holding up the game and their game timer reaches 0, they lose.

## Reconnection
<!-- SECTION_REF: 07-match-server-protocol.s011 -->
Section Ref: `07-match-server-protocol.s011`

Client reconnect request:

```ts
interface ReconnectRequest {
  type: 'reconnect';
  protocolVersion: string;
  matchId: MatchId;
  playerId: PlayerId;
  lastSeenStateSeq?: number;
  sessionToken: string;
}
```

Server behavior:

1. Authenticate session.
2. Check in-memory match.
3. If missing, attempt Redis recovery.
4. Send current filtered view.
5. Re-send pending decision if any.
6. Restore timer state.

## Disconnect handling
<!-- SECTION_REF: 07-match-server-protocol.s012 -->
Section Ref: `07-match-server-protocol.s012`

Use configurable policies:

```ts
interface DisconnectPolicy {
  gracePeriodMs: number;
  forfeitAfterMs: number;
  pauseTimersDuringGrace: boolean;
  allowReconnectAfterForfeit: boolean;
  countsAsLossOnGraceExpiry: boolean;
}
```

Recommended:

- Unranked: moderate or longer grace period; pausing the game timer during grace is acceptable.
- Ranked: shorter grace period, grace expiry becomes a disconnect forfeit, and the result feeds Elo/discipline policy.
- Custom: host-configurable within allowed bounds; passworded lobbies still use the same reconnect/auth rules.

## Timers
<!-- SECTION_REF: 07-match-server-protocol.s013 -->
Section Ref: `07-match-server-protocol.s013`

Use one primary game timer per player plus a disconnect grace timer. Do **not** create a separate mulligan timer or decision timer.

The rule is simple:

- the timer only drains for the player currently holding up progress
- if the engine is resolving mandatory work, no player timer drains
- if neither player is currently holding up the game, no player timer drains
- if a player's game timer reaches 0, that player loses immediately

Mulligan example:

- if neither player has chosen keep/mulligan yet, neither timer drains
- once one player has chosen, the other player is now holding up setup, so only that player's timer drains

```ts
interface PlayerGameTimer {
  playerId: PlayerId;
  remainingMs: number;
  isRunning: boolean;
}

interface TimerState {
  drainingPlayerId?: PlayerId;
  players: Record<PlayerId, PlayerGameTimer>;
  disconnect?: {
    playerId: PlayerId;
    startedAt: string;
    expiresAt: string;
  };
}
```

## Rate limiting
<!-- SECTION_REF: 07-match-server-protocol.s014 -->
Section Ref: `07-match-server-protocol.s014`

Rate limits should be action-aware.

| Action type | Suggested behavior |
|---|---|
| Normal action | Reject bursts above threshold |
| Decision response | Strict pending-decision match required |
| Resync request | Cooldown per client |
| Ping/pong | Standard heartbeat |
| Concede | Always allow, but dedupe |

Rejected action spam is logged for moderation.

## Match isolation
<!-- SECTION_REF: 07-match-server-protocol.s015 -->
Section Ref: `07-match-server-protocol.s015`

Every match runs inside an error boundary.

If one match crashes:

1. Freeze that match.
2. Preserve state/action logs.
3. Notify players with `matchError`.
4. Offer recovery if safe.
5. Do not crash other matches.

## Horizontal scaling
<!-- SECTION_REF: 07-match-server-protocol.s016 -->
Section Ref: `07-match-server-protocol.s016`

Early:

- One match-server instance.
- Redis persistence still enabled.

Later:

- Route by `matchId` sticky session.
- Store active match owner in Redis.
- Allow another instance to recover if owner disappears.

```ts
interface MatchOwnershipRecord {
  matchId: MatchId;
  serverInstanceId: string;
  heartbeatAt: string;
  recoverable: boolean;
}
```

## Spectator subscriptions
<!-- SECTION_REF: 07-match-server-protocol.s017 -->
Section Ref: `07-match-server-protocol.s017`

Spectator clients send no actions. They subscribe to match views under a configured spectator policy.

```ts
interface SpectatorSubscribeRequest {
  type: 'spectate';
  matchId: MatchId;
  spectatorId?: PlayerId;
  requestedMode?: string;
}
```

The server selects the allowed mode from match config and user permissions.

## Protocol compatibility
<!-- SECTION_REF: 07-match-server-protocol.s018 -->
Section Ref: `07-match-server-protocol.s018`

Each message includes `protocolVersion`. Breaking changes require either:

- Client/server deployed together, or
- Version negotiation and compatibility shims.

For early development, synchronized deployment is fine. For public users, incompatible client builds should get a clear reload prompt.
